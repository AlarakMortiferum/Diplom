# План проверки и автоматизации приложения «Мобильный хоспис»

## 1. Цель работы
Провести ручное и автоматизированное тестирование Android-приложения «Мобильный хоспис», подготовить чек-лист, тест-кейсы, UI-автотесты и итоговые отчётные артефакты для дипломного проекта.

## 2. Объект тестирования
В приложении реализован следующий функционал, подлежащий проверке:
- экран-заставка;
- экран авторизации;
- главный экран;
- раздел `News`;
- экран `Control panel` для работы с новостями;
- создание и редактирование новости;
- фильтрация новостей;
- экран `About`;
- раздел `Our Mission`;
- боковое меню и основная навигация.

Тестовые данные для входа:
- логин: `login2`
- пароль: `password2`

## 3. Тестовая среда и устройства
### Используемая среда разработки
- ОС: Windows;
- среда разработки: Android Studio / IntelliJ IDEA Community Edition **2025.2.3**;
- Gradle: **7.6**;
- JDK: **17**;
- тип сборки: `debug`.

### Фактически используемые виртуальные устройства
На момент подготовки первого этапа ручное тестирование и исследовательская проверка приложения выполнялись на следующих эмуляторах Android Studio:
- **Medium Phone API 36.1** — Android 16 (`Baklava`), `x86_64`, тип устройства: `Virtual`;
- **Pixel 9 Pro Fold API 36.0** — Android 16 (`Baklava`), `x86_64`, тип устройства: `Virtual`;
- **Pixel 4 API 29** — Android 10 (`Q`), `x86`, тип устройства: `Virtual`.

## 4. Используемые инструменты и фреймворки
### Для ручной проверки
- Android Studio;
- Android Emulator;
- IntelliJ IDEA Community Edition **2025.2.3** как установленная рабочая IDE JetBrains;
- встроенные средства анализа: `Logcat`, `Build Output`, `Device Manager`, `Layout Inspector` при необходимости.

### Для автоматизации
- **Espresso** — основной фреймворк UI-автоматизации внутри приложения;
- **UI Automator** — для сценариев, где потребуется работа с системным интерфейсом или внешними окнами Android;
- **JUnit 4** — запуск и организация тестов;
- **AndroidX Test** (`runner`, `rules`) — инфраструктура запуска Android UI-тестов;
- **Allure** — формирование отчётности по автотестам.

## 5. Границы тестирования

### Входит в тестирование
- отображение экранов-заставок и переход на экран авторизации;
- успешная и неуспешная авторизация;
- открытие главного экрана;
- просмотр списка новостей;
- раскрытие и сворачивание карточек новостей;
- переход на экран `Control panel`;
- создание и редактирование новости;
- фильтрация новостей;
- открытие экрана `About`;
- открытие раздела `Our Mission`;
- переходы через боковое меню и верхние иконки;
- проверка обязательных полей и обработки пользовательского ввода;
- базовые нефункциональные проверки интерфейса и стабильности.

### Не входит в тестирование
- тестирование серверной части и API вне пользовательского интерфейса;
- нагрузочное тестирование под высокой нагрузкой;
- анализ сетевого трафика и проверка безопасности на уровне перехвата запросов;
- проверка приложения на большом количестве различных устройств;
- полная проверка всех системных разрешений Android.

## 6. Виды тестирования
- функциональное;
- UI-тестирование;
- позитивное;
- негативное;
- регрессионное;
- исследовательское;
- базовые нефункциональные проверки.

## 7. Этапы работы

### Этап 1. Ручная проверка
На первом этапе выполняются:
1. первичный анализ приложения и его экранов;
2. smoke-проверка запуска и основных пользовательских сценариев;
3. составление чек-листа;
4. составление тест-кейсов;
5. фиксация фактических результатов и обнаруженных дефектов.

### Этап 2. Автоматизация и отчётность
На втором этапе выполняются:
1. настройка окружения для UI-автотестов;
2. подготовка структуры `app/src/androidTest`;
3. реализация автотестов по приоритетным сценариям;
4. подключение Allure и подготовка отчётов;
5. оформление файлов `README.md`, `Result.md` и `allure-results.zip`.

Автоматизация и подготовка отчётности начинаются **после проверки и одобрения первого этапа дипломным руководителем**.

## 8. Интервальная оценка трудозатрат (с учётом рисков)
### Этап 1. Ручное тестирование и документация
- первичная разведка приложения: **3–4 часа**;
- ручной smoke и ключевые проверки: **5–7 часов**;
- оформление чек-листа: **3–4 часа**;
- оформление тест-кейсов: **6–8 часов**.

**Итого по этапу 1:** **17–23 часа**.

### Этап 2. Автоматизация и отчётность
- подготовка структуры автотестов и окружения: **4–6 часов**;
- реализация автотестов по ключевым сценариям: **14–20 часов**;
- подключение Allure и проверка отчётности: **4–6 часов**;
- оформление README и Result: **3–4 часа**.

**Итого по этапу 2:** **25–36 часов**.

### Резерв на риски
- нестабильные элементы интерфейса;
- дефекты, блокирующие сценарии;
- проблемы с локаторами и ожиданиями;
- нестабильность эмулятора и пересборок;
- различия поведения интерфейса на разных форм-факторах устройств.

**Резерв:** **6–10 часов**.

### Общая оценка
**Полный объём работы по диплому:** **48–69 часов**.

## 9. Подход к ручному тестированию
1. Проверить корректность сборки и запуска приложения.
2. Пройти основные пользовательские сценарии от начала до конца.
3. Зафиксировать фактический результат по каждому пункту.
4. Отразить результаты в файлах `Check` и `Cases`.
5. Зафиксировать выявленные дефекты и блокировки.

## 10. Подход к автоматизации
После одобрения первого этапа планируется автоматизировать приоритетные и стабильные сценарии:
- переход с экрана-заставки на экран авторизации;
- успешную авторизацию;
- негативные сценарии авторизации;
- открытие главного экрана;
- открытие раздела `News`;
- переход на экран `About`;
- переход в раздел `Our Mission`;
- раскрытие карточки новости;
- открытие экрана фильтрации;
- открытие экрана `Control panel`;
- выход из аккаунта.

Основной прогон UI-автотестов планируется выполнять на **Pixel 4 API 29**, так как именно эта конфигурация соответствует требованиям дипломного проекта. При необходимости отдельные сценарии дополнительно проверяются на эмуляторах с API 36 для сравнения поведения на более новых версиях Android.

Сценарии создания/редактирования новости, которые сейчас блокируются дефектами, будут либо автоматизированы после исправления, либо отмечены как непокрытые с указанием причины.

## 11. Предварительно найденные риски и дефекты
- **BUG-001**: из раздела `News` пункт меню `About` визуально выбирается, но экран `About` не открывается;
- **BUG-002**: на экранах создания и редактирования новости поле `Description` не принимает ввод;
- **BUG-003**: требуется дополнительная проверка валидации пользовательского ввода и дат в форме новости;
- **BUG-004**: в проекте присутствуют предупреждения сборки из-за устаревших компонентов, но они не блокируют запуск приложения и выполнение диплома.
